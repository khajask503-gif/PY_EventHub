# Booking Management Test Scenarios

These scenarios are generated by the **Senior Functional Test Designer** agent, grounded in the EventHub domain rules and verified against the actual source code implementation.

### TC-001: View list of all active bookings
**Category**: Happy Path
**Priority**: P0
**Preconditions**: User is logged in and has at least 1 active booking.
**Steps**:
1. Navigate to the bookings page (`/bookings`).
2. Observe the list of bookings.
**Expected Results**: The bookings list is displayed. Each card shows the event title, booking reference, quantity, total price, and a "confirmed" status badge.
**Business Rule**: User can view bookings via "View My Bookings" link.
**Code Trace**: `frontend/components/bookings/BookingCard.jsx`
**Suggested Layer**: E2E

### TC-100: Verify booking reference format
**Category**: Business Rule
**Priority**: P0
**Preconditions**: User is logged in and is on an event booking form (e.g., "Tech Summit").
**Steps**:
1. Complete a booking for the event.
2. Observe the generated booking reference.
**Expected Results**: Reference follows the pattern `[FIRST_LETTER]-[6_RANDOM_ALPHANUMERIC]`. For "Tech Summit", it must start with "T-".
**Business Rule**: Reference first character MUST match the event title's first character (uppercase).
**Code Trace**: `backend/src/services/bookingService.js:9` (`randomRef` function)
**Suggested Layer**: API / Unit

### TC-101: Verify FIFO pruning when exceeding the 9-booking limit
**Category**: Business Rule
**Priority**: P0
**Preconditions**: User has exactly 9 active bookings. Record the ID of the oldest booking (Booking A).
**Steps**:
1. Create a new 10th booking (Booking B).
2. Fetch the list of all bookings for the user.
**Expected Results**: Total booking count remains 9. Booking B exists. Booking A has been automatically deleted.
**Business Rule**: Max 9 bookings per user. FIFO pruning enforced on 10th booking.
**Code Trace**: `backend/src/services/bookingService.js:62` (`MAX_USER_BOOKINGS`)
**Suggested Layer**: API
